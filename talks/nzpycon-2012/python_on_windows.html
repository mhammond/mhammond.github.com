Python on Windows

CPython on Windows

  • First-class support for Windows. Eg: Unicode APIs are preferred to offer seamless Unicode experience.
  • Tends to avoid platform-specific functionality
    Native windows functions are used to create a cross-platform environment

CPython on Windows

  • Not much Windows specific functionality is exposed.
    Just Windows specific implementations of cross-platform features.
  • ctypes or pywin32 is often used to fill in the gaps.
Much of CPython uses unicode windows apis

Leveraged by many other projects;

Cross-platform?

Python has rich cross-platform support - so why target Windows specific features?

  • Provide good experience for administrators.
  • Provide good experience for users.
  • Integrate with other languages and development environments.
So if Python is cross-platform, why do I care about Windows specific features?

For administrators

  • NTLM/Kerberos
    admins like the security of not providing passwords to applications.
  • Integrate with existing user roles
    admins use Windows tools to grant access
  • Implement 'daemon' as a service
    give administrators a single control point.
  • Integrated performance monitoring, event logging, etc

For users

  • NTLM/Kerberos for convenience
  • Use native widgets for navigating the file-system and shell namespace
  • Leverage metadata stored in files
  • Access printers, USB devices, etc

Integrate with development tools

  • Some shops have standardized on Windows, not Python
  • Use Visual Studio as a development environment.
  • Write reusable components in languages other than Python

Windows Management Instrumentation (WMI)

  • A (somewhat) consistent interface to common management tasks.
  • Start a process on a remote computer
  • Schedule a process
  • Reboot a computer
  • Query event log, etc

Windows Management Instrumentation (WMI)

But can't all these be done without WMI?

Yes, but only with various low-level API calls.

Which is the fallback if WMI can't do it...

WMI example

Show all automatic services which are not running

import wmi
c = wmi.WMI()
stopped = c.Win32_Service(StartMode="Auto", 
                          State="Stopped")
for s in stopped:
    print s.Caption, "service is not running"

Python launcher - why now?

Previously 'last version installed wins'

  • Not really a problem with 2.x series.

Python 3.x made this a real problem.

  • Each script is tied to a specific version.

Launcher Shebang processing

Compatible with *nix shebang lines.
#! /usr/bin/python
#! /usr/bin/env python
Windows specific shebang lines.
#! c:\python27\python.exe Minimal shebang lines
#! python3

Version specific shebang lines

Symlinks are often used on Linux to specify a version.
#! /usr/bin/python3

Launcher handles this too
#! /usr/bin/python3
#! /usr/bin/python2.7

Launcher on the command-line

c:\> py script.py
Launches the latest installed Python 2.x - No need for them all to be on the PATH

c:\> py -3.2 script.py
c:\> py -3.3-32 script.py
Launch 3.2, 32bit 3.3

The 64bit version of Python is always preferred if both are found. -32 allows you to override this.

Full Unicode Implementation in 3.x

All use of Windows API by core Python uses Unicode

Implementation detail which is completely invisible, but still nice!

Optionally add Python to the PATH

  • New installer option to add Python to the PATH.
  • Off by default - hard to get right with multiple versions.
  • Less relevant now the launcher exists

Links

  • This probably means it is less useful than it sounds.
